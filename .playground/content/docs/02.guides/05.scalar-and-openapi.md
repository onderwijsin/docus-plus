Docus Plus includes a Scalar API Explorer that is configured by environment variables. Once an
OpenAPI source is configured, the layer automatically enables Scalar, loads the document, and
creates searchable API content. You do not need to write a Markdown page for every endpoint.

## Enable the integration

Add both variables to the environment profile used for the build:

```dotenv [envs/.env.development]
OPENAPI_SOURCE_TYPE=remote
OPENAPI_LOCATION=https://api.example.com/openapi.json
```

The source type is either `remote` or `local`:

| Source   | `OPENAPI_LOCATION`     | Use when                                                      |
| -------- | ---------------------- | ------------------------------------------------------------- |
| `remote` | An HTTPS URL           | The OpenAPI document is hosted by an API registry or service. |
| `local`  | A path below `public/` | The specification is committed with the documentation site.   |

For a local specification, put the file below `public/` and use its public path:

```dotenv [envs/.env.production]
OPENAPI_SOURCE_TYPE=local
OPENAPI_LOCATION=openapi/production.json
```

The file is then available to the browser at `/openapi/production.json`. The local file must stay
inside `public/`; paths that escape that directory are rejected.

Both variables are required to enable the integration. If either is missing, the layer leaves the
API reference disabled and the `/api-reference` route redirects to the documentation home page.

## Use the API Explorer

When enabled, the interactive Scalar Explorer is available at:

```text
/api-reference
```

The layer adds an API Explorer button to the header and an API Reference entry to global search.
Scalar is configured with the selected document and uses the same application color mode. The
playground's [API reference](/api-reference) is enabled by its remote OpenAPI profile.

Consumers can still extend the Scalar configuration from `nuxt.config.ts` when they need a Scalar
option that is not covered by the shared defaults:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scalar: {
    showSidebar: true,
    hideModels: false
  }
});
```

The integration keeps the API Explorer at `/api-reference` and supplies the configured OpenAPI
URL. Use [Scalar's Nuxt documentation](https://scalar.com/products/api-references/integrations/nuxt)
for additional Scalar-specific options.

## What gets parsed at build time

The OpenAPI document is fetched or read while Nuxt Content prepares its collections. The layer then:

1. Validates the document with Scalar's OpenAPI parser.
2. Normalizes the validated specification so supported OpenAPI versions have a consistent shape.
3. Reads the document information and servers.
4. Creates virtual content entries for the API introduction, tags, operations, and schemas.
5. Extracts useful endpoint and model text for full-text search.
6. Generates a Scalar URL for each entry so search results open the matching Explorer view.

These entries exist in the Nuxt Content database and are kept in memory during generation. They do
not create source Markdown files in the consuming repository. The generated content powers API
search and the layer's API-aware assistant context in addition to the interactive Scalar page.

The parser also follows local `$ref` references when collecting operation and schema text. This
means parameters, request bodies, responses, and nested model properties can be found through
search even when the specification uses reusable components.

## OpenAPI document requirements

Remote documents must use HTTPS and must be reachable during the Nuxt build. Local documents must
be below `public/`. The specification must pass validation, and external `$ref` files are not
resolved automatically. Bundle external references into one document before building:

```text
OpenAPI document
├── info, servers
├── paths
│   └── operations, parameters, request/response schemas
└── components
    └── schemas and local #/ references
```

If parsing fails, the build reports the source URL or path and the validation errors. Fix the
specification or the selected profile before deploying; an incomplete API reference should not be
silently published.

## Choose a document per environment

Use different profiles when preview and production expose different API contracts:

```dotenv [envs/.env.preview]
MODE=preview
OPENAPI_SOURCE_TYPE=remote
OPENAPI_LOCATION=https://preview-api.example.com/openapi.json
```

```dotenv [envs/.env.production]
MODE=production
OPENAPI_SOURCE_TYPE=remote
OPENAPI_LOCATION=https://api.example.com/openapi.json
```

The profile selected at build time determines both the document loaded by the parser and the URL
used by Scalar in the generated application. Keep the source URL in the profile rather than
hard-coding environment-specific values in shared layer code.

::tip{title="No manual API page is required"}
Add `OPENAPI_SOURCE_TYPE` and `OPENAPI_LOCATION`, run the normal Nuxt build or generate command,
and the API Explorer, generated API content, and search records are created automatically.
::
